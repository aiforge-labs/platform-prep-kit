## 1. Terraform State Management

### What the state file contains
- Resource mappings: Terraform resource ID → cloud resource ID
- Metadata: resource dependencies, provider versions
- Sensitive outputs: credentials, private keys (treat the state file as sensitive)

### Remote backend comparison
| Backend | Lock mechanism | Best for |
|---|---|---|
| S3 + DynamoDB | DynamoDB conditional writes | AWS-native teams |
| Azure Blob + Table | Azure Blob lease | Azure-native teams |
| GCS | GCS object lock | GCP-native teams |
| Terraform Cloud/Enterprise | API-based | Any, managed service |

### State locking
Prevents concurrent applies. Always use a locking backend in team environments. Force-unlock only when you are certain no apply is running:
```bash
terraform force-unlock <LOCK_ID>
```

### State manipulation (use with caution)
```bash
terraform state list               # list all resources in state
terraform state show <resource>    # show state for a resource
terraform state mv <src> <dst>     # rename/move resource in state (for refactoring)
terraform state rm <resource>      # remove from state (stops tracking, does NOT destroy)
terraform import <resource> <id>   # import existing resource into state
```

### Handling drift
Drift = resource modified outside Terraform. Detect with:
```bash
terraform plan  # shows changes needed to match state
```
Resolution options: (1) update Terraform config to match reality, (2) run apply to restore declared state, (3) use `terraform import` if state is missing.

## 2. Terraform Module Design Patterns

### Standard module structure
```
modules/
  rds-postgres/
    main.tf          # resource definitions
    variables.tf     # input variables with descriptions and validation
    outputs.tf       # outputs for consumers
    versions.tf      # required providers and versions
    README.md        # auto-generated by terraform-docs
```

### Input variable design
```hcl
variable "instance_class" {
  type        = string
  description = "RDS instance class"
  default     = "db.t3.medium"

  validation {
    condition     = can(regex("^db\\.", var.instance_class))
    error_message = "Instance class must start with 'db.'"
  }
}
```

### Module versioning
Pin modules to specific versions to prevent breaking changes:
```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"  # allows patch updates, blocks major updates
}
```

### Terragrunt DRY pattern
Instead of duplicating provider config across environments:
```hcl
# terragrunt.hcl (root)
remote_state {
  backend = "s3"
  config = {
    bucket = "my-terraform-state"
    key    = "${path_relative_to_include()}/terraform.tfstate"
    region = "us-east-1"
  }
}

# environments/prod/vpc/terragrunt.hcl
include "root" {
  path = find_in_parent_folders()
}

terraform {
  source = "../../../modules//vpc"
}

inputs = {
  cidr = "10.0.0.0/16"
}
```

## 3. GitOps Pull vs Push Model

See also: `platform-engineering.md` for ArgoCD vs Flux comparison.

### Multi-environment promotion with ArgoCD ApplicationSets

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: my-app
spec:
  generators:
    - list:
        elements:
          - cluster: dev
            url: https://dev.k8s.example.com
          - cluster: staging
            url: https://staging.k8s.example.com
          - cluster: prod
            url: https://prod.k8s.example.com
  template:
    metadata:
      name: "my-app-{{cluster}}"
    spec:
      project: default
      source:
        repoURL: https://github.com/myorg/my-app
        path: "environments/{{cluster}}"
      destination:
        server: "{{url}}"
        namespace: my-app
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
```

### Environment promotion pipeline pattern
```
feature-branch → dev (auto-deploy) → staging (auto-deploy + smoke test) → prod (manual approval)
```
Each environment has its own overlay directory:
```
environments/
  dev/
    kustomization.yaml  (overrides: replicas=1, image=latest)
  staging/
    kustomization.yaml  (overrides: replicas=2, image=v1.2.3-rc1)
  prod/
    kustomization.yaml  (overrides: replicas=5, image=v1.2.3)
```

## 4. IaC Testing Pyramid

```
         ┌───────────────────┐
         │   End-to-End      │  Real infra deployed + tested (Terratest)
         ├───────────────────┤
         │   Integration     │  Module deployed, outputs validated
         ├───────────────────┤
         │   Static/Unit     │  terraform validate, tflint, Checkov, OPA/Conftest
         └───────────────────┘
```

### Static analysis tools
| Tool | What it checks | Language |
|---|---|---|
| `terraform validate` | HCL syntax and internal consistency | Built-in |
| `tflint` | Best practices, unused variables, deprecated syntax | HCL |
| Checkov | Security and compliance misconfigurations | Multi (Python rules) |
| tfsec | Terraform-specific security issues | Terraform |
| Trivy | Multi-framework IaC scanning | Multi |
| Conftest + OPA | Custom policy validation against Terraform plan JSON | Rego |

### Conftest for Terraform plan validation
```bash
# Generate plan as JSON
terraform plan -out=plan.bin && terraform show -json plan.bin > plan.json

# Run OPA policies against the plan
conftest test plan.json --policy policies/
```

Example OPA policy (deny unencrypted S3 buckets):
```rego
package main

deny[msg] {
  resource := input.resource_changes[_]
  resource.type == "aws_s3_bucket"
  resource.change.after.server_side_encryption_configuration == null
  msg := sprintf("S3 bucket '%v' must have server-side encryption enabled", [resource.address])
}
```

### Terratest example structure
```go
func TestVpcModule(t *testing.T) {
    t.Parallel()

    terraformOptions := &terraform.Options{
        TerraformDir: "../modules/vpc",
        Vars: map[string]interface{}{
            "cidr": "10.0.0.0/16",
            "name": "test-vpc",
        },
    }

    defer terraform.Destroy(t, terraformOptions)
    terraform.InitAndApply(t, terraformOptions)

    vpcId := terraform.Output(t, terraformOptions, "vpc_id")
    assert.NotEmpty(t, vpcId)
}
```

## 5. Policy-as-Code Enforcement Layers

| Layer | Tool | Timing | Type |
|---|---|---|---|
| Local dev | pre-commit hooks (tfsec, checkov) | Before commit | Preventive |
| PR/CI | Checkov, Conftest, tfsec | Before merge | Preventive |
| Plan validation | OPA/Conftest against plan JSON | Before apply | Preventive |
| Runtime | AWS Config, Azure Policy, GCP Org Policy | After deploy | Detective |
| Kubernetes | OPA Gatekeeper, Kyverno | At admission | Preventive |

Defense-in-depth means policies exist at multiple layers. Preventive (block before it happens) is always better than detective (alert after the fact).

## 6. Atlantis GitOps Workflow

```
Developer opens PR
       ↓
Atlantis detects Terraform changes
       ↓
atlantis plan (auto or manual comment)
       ↓
Plan output posted as PR comment
       ↓
Reviewers see exactly what will change
       ↓
PR merged
       ↓
atlantis apply (auto-merge or manual comment)
       ↓
Terraform apply runs, output posted
```

### atlantis.yaml example
```yaml
version: 3
projects:
  - name: networking
    dir: environments/prod/networking
    workspace: prod
    autoplan:
      when_modified: ["*.tf", "*.tfvars"]
    apply_requirements: [mergeable, approved]
  - name: compute
    dir: environments/prod/compute
    workspace: prod
    autoplan:
      when_modified: ["*.tf"]
    apply_requirements: [mergeable, approved]
```

## 7. Multi-Account / Multi-Environment Patterns

### AWS Organizations structure
```
Root
├── Security OU
│   └── Security account (GuardDuty delegated admin, Security Hub)
├── Infrastructure OU
│   ├── Shared services account (DNS, ECR, artifact registry)
│   └── Network account (Transit Gateway, VPCs)
├── Workloads OU
│   ├── Dev account
│   ├── Staging account
│   └── Prod account
└── Sandbox OU
    └── Developer sandbox accounts
```

### Backend configuration per environment
```hcl
# environments/prod/backend.tf
terraform {
  backend "s3" {
    bucket  = "my-tfstate-prod"
    key     = "networking/terraform.tfstate"
    region  = "us-east-1"
    encrypt = true
    dynamodb_table = "terraform-locks"
  }
}
```

### Environment promotion strategy
- **Modules** have their own versions (semver). Environments pin to specific versions.
- Promote: dev → staging → prod by bumping the module version in each environment's config.
- Breaking changes in modules get a major version bump.
- Changelogs are maintained per module.
